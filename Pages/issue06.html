SMB Signing Mismatch ‚Äì Session Setup Failure

Layer: Session (L5)

Symptom: Windows clients cannot map \\FILESRV\Share ‚Äì error STATUS_ACCESS_DENIED; Wireshark shows ‚ÄúSMB2 SESSION_SETUP ‚Üí STATUS_USER_SESSION_DELETED‚Äù

Root Cause: Server enforces mandatory SMB signing, while clients negotiate unsigned SMB2 sessions; the session fails during setup handshake

Objective

  • Identify the signing-mismatch with packet capture and event logs
  • Align server and client Signing policies (either enable on client or relax on server)
  • Restore successful SMB sessions and drive mappings

Baseline Topology

‚Ä¢ FileSrv: Windows Server 2022 DC, SMB Signing = Required
‚Ä¢ Client-A: Windows 10 22H2 laptop, on-prem AD
‚Ä¢ Network: 1 Gbps campus LAN, no middleboxes modifying SMB
