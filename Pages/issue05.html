TCP Window-Scaling Mismatch ‚Äì Throughput Collapse

Layer: Transport (L4)

Symptom: Long-haul file transfers stall at <300 kB/s; Wireshark shows Window Size = 0 + repeated ACKs

Root Cause: One endpoint advertises RFC 7323 window-scale = 10, the middlebox strips the option; the peer believes scale = 0 and shrinks the receive window ‚ü∂ 0-window condition

Objective

  • Detect the scaling-option mismatch with packet capture
  • Apply a firewall policy that preserves TCP options (or selectively disables scaling)
  • Restore full 1 Gbit throughput on WAN transfers

Baseline Topology

‚Ä¢ Server-A (Ubuntu 22.04, 10.0.10.10) in DC-1
‚Ä¢ Client-B (Windows 11, 192.0.2.50) in HQ
‚Ä¢ MPLS WAN ‚Üí stateful firewall (FortiGate 200F, SW 7.2) ‚Üí Core switch
‚Ä¢ RTT ‚âà 60 ms, 1 Gbps link capacity

Step 1 ‚Äì Capture & Analyse

PS> netsh trace start capture=yes ipv4.address=10.0.10.10
# start 2 GB upload via scp ‚Ä¶ stalls at 4 MB
PS> netsh trace stop
# Open ETL in Wireshark
#   ‚ûú SYN from B:  window 64240, wscale 10
#   ‚ûú SYN-ACK:     wscale 7
#   ‚ûú ACK:         no wscale option (!)

Step 2 ‚Äì Firewall Fix (preserve TCP options)

config firewall profile-protocol-options
 edit "WAN-TCP"
  config tcp
   set status enable
   set options no-strip      <-- keep WS/SACK/MSS
  end
 next
end
#
config firewall policy
 edit 10
  set name "DC1-HQ"
  set profile-protocol-options "WAN-TCP"
 next
end

Alternative ‚Äì Disable Scaling on the Server

(Temporary mitigation if firewall cannot be changed)

# Ubuntu Server-A
echo "net.ipv4.tcp_window_scaling = 0" | sudo tee -a /etc/sysctl.d/disable-wscale.conf
sudo sysctl -p /etc/sysctl.d/disable-wscale.conf

  1. Repeat the 2 GB SCP upload ‚Üí sustained >900 Mbps, no 0-window stalls.
  2. Wireshark: ACK now contains WScale=10 in every handshake leg.
  3. Firewall logs show zero ‚ÄúTCP option stripped‚Äù events.
